# Markdown 文档知识库使用教程与项目亮点

## 一、项目简介

这是一个面向日常查阅、技术文档管理、知识沉淀和轻量编辑的本地 Markdown 文档知识库系统。它可以自动读取项目目录中的 `.md` 文件，按照文件夹结构生成清晰的目录树，并提供全文搜索、文档预览、在线编辑、截图粘贴、知识图谱、文件管理等能力。

相比传统的文件夹加编辑器模式，本项目更像一个专注于 Markdown 文档的本地知识工作台：打开即可浏览，搜索即可定位，拖动即可整理，编辑即可保存，图谱即可观察文档之间的关联关系。

## 二、适合哪些使用场景

本项目适合以下场景：

- 个人技术笔记管理
- 项目说明文档维护
- 部署流程、排障记录、运维手册整理
- 学习资料、知识卡片、方案文档归档
- 团队内部轻量知识库
- 本地 Markdown 文件的快速检索与阅读
- 带截图的操作教程、测试记录、问题复盘

如果你经常在多个 Markdown 文件之间切换，或者希望把零散文档整理成一个可搜索、可编辑、可关联的知识系统，这个项目会非常适合。

## 三、如何启动项目

### 1. 进入项目根目录

在终端中进入项目所在目录：

```powershell
cd E:\codex\mytemple
```

### 2. 启动服务

项目已经配置了 `npm start` 命令，可以直接执行：

```powershell
npm start
```

也可以直接使用 Node 启动：

```powershell
node server.js
```

启动成功后，终端会显示类似信息：

```text
Markdown knowledge app running at http://localhost:4173
```

### 3. 打开浏览器访问

在浏览器中打开：

```text
http://localhost:4173/
```

打开后即可看到 Markdown 文档知识库界面。

### 4. 修改启动端口

默认端口是 `4173`。如果该端口被占用，可以临时指定其他端口：

```powershell
$env:PORT=4180
npm start
```

然后访问：

```text
http://localhost:4180/
```

### 5. 放置文档的位置

项目会自动扫描 `docs` 目录下的 Markdown 文件：

```text
E:\codex\mytemple\docs
```

把 `.md` 文件放入这个目录，刷新页面后即可在左侧目录树中看到。截图和粘贴图片会保存到项目根目录下的 `source` 文件夹。

## 四、核心功能教程

### 1. 浏览 Markdown 文档

启动项目后，系统会自动扫描 `docs` 目录下的 Markdown 文件，并在左侧生成目录树。

你可以：

- 点击任意文件查看内容
- 展开或折叠文件夹
- 按文件夹分类浏览文档
- 在不同文档之间快速切换

文档会以 Markdown 格式渲染显示，适合阅读技术说明、部署文档、开发记录和知识笔记。

### 2. 全局关键词搜索

左侧搜索框支持全局搜索。输入关键词后，系统会在所有 Markdown 文档中检索匹配内容，并展示相关结果。

搜索适合用于：

- 快速定位某个命令
- 查找接口说明
- 查找错误信息
- 查找某个配置项
- 在大量文档中追踪主题

搜索结果会帮助你快速跳转到相关文档，减少手动翻找时间。

### 3. 新建文件夹和 Markdown 文件

系统支持直接在界面中新建文件夹和 Markdown 文件，不需要频繁切回资源管理器。

你可以：

- 新建分类文件夹
- 在指定目录下创建 `.md` 文件
- 快速建立新的笔记或技术文档
- 按项目、模块、主题整理资料

这让文档管理更接近日常知识库的使用方式。

### 4. 删除文件

当文档不再需要时，可以在界面中执行删除操作。删除前会进行确认，避免误操作。

这个功能适合清理：

- 临时记录
- 重复文档
- 过期说明
- 测试文件

### 5. 拖动整理目录

项目支持拖动文件和文件夹来整理目录结构。

你可以：

- 将任意 Markdown 文件拖动到任意文件夹
- 将文件拖回根目录
- 将文件夹拖动到其他文件夹下
- 调整文档的分类归属

这项功能让知识整理更加直观，适合在文档数量逐渐增多后进行重新归类。

### 6. 编辑 Markdown 文档

系统提供编辑模式，可以直接修改当前 Markdown 文档。

编辑能力包括：

- 普通文本编辑
- 加粗
- 字号调整
- 文字颜色
- 背景高亮
- 下划线
- 删除线
- 分割线
- 重点标注
- 撤销操作，支持 `Ctrl + Z`

它既保留 Markdown 的轻量特性，也提供了常用的富文本式辅助操作，适合写教程、整理方案、记录重点内容。

### 7. 截图粘贴、存储与展示

项目重点支持截图型文档写作。你可以把软件界面、报错信息、配置页面、操作步骤、测试结果等截图直接粘贴到 Markdown 文档中，系统会自动把图片保存到根目录下的 `source` 文件夹，并在当前文档里插入可展示的图片引用。

使用方式：

1. 截图后复制图片。
2. 在编辑模式中粘贴。
3. 系统自动生成图片文件。
4. 文档中自动插入图片路径。
5. 回到预览模式后，截图会作为文档内容直接展示。

这个能力让 Markdown 文档不再只是文字记录，而是可以形成完整的图文教程。对技术文档来说，截图往往比大段文字更直观：一个配置界面、一条报错提示、一个部署成功页面，都能让读者快速理解当前步骤。

图片会以适合文档阅读的方式显示，并考虑压缩存储，减少截图文件对磁盘空间的占用。文档内容和图片资源分开保存，正文保持清爽，图片统一归档，后续迁移、备份和清理都更方便。

这个功能非常适合编写：

- 操作教程
- 软件安装说明
- 测试过程记录
- 问题复现步骤
- UI 修改说明
- 项目演示文档
- 部署成功截图归档
- 报错排查和解决方案复盘
- 客户或团队汇报材料

例如，在编写部署教程时，可以一边记录命令，一边粘贴终端输出截图、浏览器访问成功截图和配置页面截图。最终形成的文档既有文字步骤，也有真实画面证据，更适合交付、复盘和二次查阅。

### 8. 知识图谱

项目提供文档知识图谱功能，可以展示 Markdown 文件之间的关联关系。

图谱关联来源包括：

- 文档链接
- 共同标签
- 相似语义
- 共享关键词

系统会把多个文档中反复出现的关键词提取为关键节点，并连接相关文档。这样你可以从图谱中看到哪些文档属于同一主题，哪些知识点在多个文档中重复出现。

这对技术文档尤其有价值，因为很多知识并不是孤立存在的。例如部署文档、架构说明、搜索设计和 README 之间，往往会共享相同的关键词和上下文。

## 五、项目亮点

### 1. 本地优先，数据可控

所有 Markdown 文档都存放在本地目录中，文档内容清晰可见，不依赖复杂数据库，也不强制绑定云服务。

这带来几个好处：

- 文档安全可控
- 迁移简单
- 备份方便
- 可直接用其他 Markdown 工具打开
- 不容易被平台锁定

### 2. Markdown 原生友好

项目围绕 Markdown 文件构建，不把文档锁进私有格式。即使不使用本系统，文件仍然是标准 `.md` 文档。

这对长期维护非常重要。好的知识库不应该让内容依赖某个工具才能打开，而应该让工具服务于内容。

### 3. 目录树清晰，适合技术资料沉淀

左侧目录树按照文件夹结构展示，天然适合技术文档的分类方式。

例如：

- `architecture` 存放架构设计
- `github` 存放部署流程
- `source` 存放图片资源
- 根目录存放常用说明

这种结构符合开发者和技术人员的日常习惯。

### 4. 搜索速度快，定位效率高

系统会读取文档内容并建立检索能力。相比一个个打开文件查找，全局搜索能快速把你带到目标文档。

当文档数量增加后，搜索能力会成为知识库最重要的效率入口。

### 5. 阅读和编辑一体化

传统方式通常需要在文件管理器、Markdown 编辑器、浏览器预览之间来回切换。本项目把查看、编辑、保存和管理集中在同一个界面中。

你可以在阅读时发现问题，立即切换到编辑模式修改，再回到预览状态继续查看。

### 6. 截图展示能力更适合做教程和交付

很多技术文档离不开截图，例如安装步骤、界面说明、报错信息、配置页面等。

本项目支持粘贴截图自动存储，并插入 Markdown 图片路径，减少手动保存图片、复制路径、调整引用的重复操作。截图会直接嵌入文档预览区域，形成清晰的图文展示效果。

这项能力特别适合把普通笔记升级成可展示、可交付的教程文档：

- 写安装教程时，可以展示每一步界面状态。
- 写问题复盘时，可以保留报错截图和解决后的对比截图。
- 写测试记录时，可以保存关键页面、结果面板和验证截图。
- 写产品说明时，可以用真实界面截图解释功能入口。
- 写部署文档时，可以展示服务启动、访问成功和配置完成的证据。

相比只写文字，图文结合的文档更容易被理解，也更适合团队成员交接、客户演示和后续回看。

### 7. 知识图谱帮助发现隐藏关系

文件夹目录解决的是“文档放在哪里”的问题，知识图谱解决的是“文档之间有什么关系”的问题。

通过图谱，你可以看到：

- 哪些文档共享相同关键词
- 哪些文档互相引用
- 哪些主题是知识库中的核心主题
- 哪些文档可能应该归类到一起

这让知识库不只是文件集合，而是逐渐形成结构化知识网络。

### 8. 界面简约，适合长期使用

项目界面以技术文档查阅为核心，没有复杂装饰，也不做花哨的营销式布局。左侧目录、主阅读区、编辑工具、图谱视图都围绕效率设计。

这种克制的界面适合长时间使用，不会干扰阅读和写作。

### 9. 轻量架构，便于后续打包

项目采用轻量化实现思路，避免引入过重依赖，为后续独立软件化和小体积打包留下空间。

相比大型 Electron 应用，轻量方案更容易控制体积、启动速度和资源占用。

## 六、推荐使用流程

### 第一步：建立文档分类

先根据自己的项目或知识领域创建几个主要文件夹，例如：

- `architecture`
- `deploy`
- `debug`
- `notes`
- `meeting`
- `source`

分类不需要一开始就完美，可以先粗略建立，再通过拖动文件逐步整理。

### 第二步：导入或新建 Markdown 文件

把已有 `.md` 文件放入 `docs` 目录，或者直接在界面中新建文件。

建议一个文档只记录一个主题，例如：

- 一个部署流程
- 一个问题复盘
- 一个接口说明
- 一个架构模块
- 一个工具教程

这样后续搜索和图谱关联会更清晰。

### 第三步：使用搜索快速定位

当文档逐渐增多后，不要只依赖目录查找。可以直接输入关键词搜索，例如：

- `github`
- `部署`
- `架构`
- `配置`
- `错误`
- `接口`

搜索可以帮助你快速回到历史记录中的某个知识点。

### 第四步：边查阅边编辑

遇到过期内容、表达不清或需要补充截图时，直接进入编辑模式修改。

建议保持文档结构清晰：

- 使用标题分层
- 使用列表表达步骤
- 使用代码块保存命令
- 使用截图说明界面
- 使用重点标注突出风险点

### 第五步：通过图谱检查知识关系

定期打开知识图谱，观察文档之间的连接关系。

如果某个关键词连接了很多文档，说明它可能是一个核心主题，可以考虑单独整理成一篇总结文档。

如果某些文档长期孤立，可能需要补充链接、标签或相关说明。

## 七、写作建议

为了让知识库越用越有价值，推荐采用以下写作习惯：

- 标题要明确，不要过度简写。
- 每篇文档开头说明用途。
- 操作步骤尽量编号。
- 命令、路径、配置项使用代码块。
- 重要风险使用加粗或高亮。
- 截图只保留必要内容。
- 文档之间可以增加 Markdown 链接。
- 定期清理过期文档。

好的知识库不是一次性写完的，而是在日常使用中持续生长的。

## 八、项目价值总结

这个 Markdown 文档知识库项目的核心价值，可以概括为四句话：

1. 用本地文件保存知识，稳定、透明、可迁移。
2. 用目录树管理结构，清晰、直观、符合技术习惯。
3. 用搜索和图谱建立连接，让文档不再孤立。
4. 用编辑和截图能力提升写作效率，让记录更自然。

它不是一个复杂笨重的平台，而是一个围绕 Markdown 文档打造的轻量知识工作台。对于需要长期维护技术文档、项目说明、部署流程和学习笔记的人来说，它能显著降低查找、整理和更新文档的成本。

如果你的项目里已经积累了大量 Markdown 文件，那么这个工具可以把它们从“散落的文件”变成“可查、可改、有关联的知识库”。
