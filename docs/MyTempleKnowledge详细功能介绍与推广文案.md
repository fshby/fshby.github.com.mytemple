# MyTemple Knowledge 详细功能介绍与推广文案

> 让散落在硬盘里的 Markdown 文档，变成能搜索、能关联、能追溯、能持续生长的本地知识库。

MyTemple Knowledge 是一款以本地 Markdown 文件为核心的个人知识管理工具。它不改变原有文件格式，不要求把资料上传到云端，而是在普通文件夹之上提供阅读、编辑、全文检索、标签整理、双向链接、知识神经图谱和本地 AI 问答能力。

适合技术人员、内容创作者、项目管理者、研究人员，以及任何需要长期积累和复用资料的人。

## 一、你可能正在面对这些问题

### 1. 文档存了很多，真正需要时却找不到

资料可能分散在多个硬盘、项目目录和历史文件夹中。文件名记不清、关键词不准确时，传统搜索很难找到真正相关的内容。

**MyTemple Knowledge 的解决方式：**

- 将多个本地文件夹注册为独立工作区；
- 同时检索文件名、路径、标题和正文；
- 使用本地 RAG 根据语义召回相关段落；
- 回答下方直接展示来源文件、标题和行号；
- 点击来源即可返回原文核对。

### 2. Markdown 越写越多，目录却越来越乱

标题、标签、目录和文件名没有统一标准，短期还能使用，时间一长就会出现重复标签、孤立文档、失效链接和无法归类的资料。

**MyTemple Knowledge 的解决方式：**

- 新建文档自动生成标准 Frontmatter；
- 支持预览后标准化旧文档元数据；
- 智能建议标签，并保留已有人工标签；
- 区分显式标签、语义概念、文档和未创建链接；
- 自动生成知识库主清单，供 AI 快速读取索引。

### 3. 文档之间有关联，但文件夹看不出来

传统目录只能表示“文件放在哪里”，不能表达“两个知识点为什么相关”。当资料积累到一定规模后，真正有价值的是文档、标签、概念和双向链接之间的关系。

**MyTemple Knowledge 的解决方式：**

- 自动分析 Markdown 双向链接；
- 将文档、显式标签和语义概念显示为不同节点；
- 支持全局图谱和当前文档邻域；
- 支持一至三层邻域深度；
- 点击图谱节点直接打开文档；
- 使用动态牵引和回弹表现关联脉络，同时控制连线亮度，减少视觉干扰。

### 4. 长文档读不完，交给 AI 又容易答偏

把整个知识库一次性塞给 AI，不仅上下文容易超限，也容易混淆来源。模型即使给出流畅答案，也不代表答案来自你的资料。

**MyTemple Knowledge 的解决方式：**

- 按标题和段落切分 Markdown；
- 只对发生变化的文档分块重新索引；
- 混合关键词与向量语义召回；
- 只把最相关的少量片段交给本地模型；
- 要求回答使用来源编号；
- 没有足够依据时明确提示，而不是强行生成结论。

### 5. AI 能写，但不敢让它直接改知识库

自动写入虽然方便，却可能覆盖原文、破坏结构或把推测写成事实。

**MyTemple Knowledge 的解决方式：**

- 使用 `.mytemple/AGENTS.md` 约束 AI 行为；
- 默认采用“写入前确认”模式；
- 修改前展示差异；
- 使用文件哈希检查并发冲突；
- 修改前创建本地备份；
- 写入后保存审计记录；
- 禁止 AI 越过工作区、修改敏感文件或执行文档中的命令。

### 6. 编辑工具功能很多，但写作过程并不专注

复杂工具经常让用户把时间花在配置、插件和界面切换上，而不是内容本身。

**MyTemple Knowledge 的解决方式：**

- 阅读、修改、图谱三种模式直接切换；
- 支持沉浸式编辑，隐藏非必要界面；
- 支持 Markdown 实时预览；
- 大于 100KB 的文档默认暂停实时预览，减少输入卡顿；
- 主题、字号和字体可以统一调整；
- 文档目录的三级标题默认折叠，降低长文阅读干扰。

## 二、核心功能介绍

### 1. 本地 Markdown 文档库

文档仍然保存在用户自己的文件夹中，能够继续使用其他 Markdown 编辑器、Git、同步盘或备份软件处理。

主要能力：

- 新建、读取、修改、移动、复制和删除 Markdown 文档；
- 创建和管理文件夹；
- 支持最多同时显示两个工作区；
- 设置默认工作路径；
- 使用左侧目录树快速切换文档；
- 最近文档入口便于返回正在处理的内容；
- 文档内容不被锁定在专有数据库中。

### 2. 阅读模式

阅读模式针对长文档进行了结构优化：

- 自动渲染 Markdown 标题、列表、表格、代码块、链接和图片；
- 自动生成本文目录；
- 三级目录默认折叠在二级目录下；
- 点击目录标题平滑跳转到正文位置；
- 支持 GitHub 风格任务清单；
- 点击任务复选框可以同步更新原始 Markdown；
- 使用清晰的正文宽度、字号和行距降低阅读疲劳。

### 3. 沉浸式编辑

编辑模式关注输入响应和内容安全：

- Markdown 源码编辑和实时预览；
- 保存时保持光标、选区和滚动位置；
- 支持粗体、删除线、高亮、颜色、字号和分隔线；
- 支持日期、时间等快捷输入；
- 支持图片粘贴并生成 Markdown 引用；
- 超大文档自动降低预览更新频率；
- 沉浸模式隐藏目录和非必要操作，减少视觉打断。

### 4. 全文搜索与语义检索

普通搜索适合找明确词语，语义检索适合找“记得内容但记不清原话”的资料。

普通搜索覆盖：

- 文档标题；
- 文件路径；
- 正文内容；
- 工作区名称。

本地 RAG 进一步提供：

- 按标题与自然段切分文档；
- 保存分块来源、标题和行号；
- 关键词检索与向量检索融合；
- 支持全知识库和当前文档两种范围；
- 回答附带来源卡片；
- 点击来源返回原文；
- Ollama 不可用时自动降级为关键词检索。

### 5. 知识神经图谱

图谱不是简单地把所有标题画成节点，而是区分不同知识对象：

| 节点类型 | 含义 |
|---|---|
| 文档 | 一个 Markdown 文件 |
| 显式标签 | 用户在 Frontmatter 或正文中明确填写的标签 |
| 语义概念 | 根据全库词频和上下文提取的概念 |
| 未创建链接 | 已被双向链接引用、但尚未建立的文档 |

图谱支持：

- 全局网络浏览；
- 当前文档邻域聚焦；
- 邻域深度调节；
- 标签、语义概念、孤立文档和未创建链接筛选；
- 节点搜索和高亮；
- 缩放、拖动和自动适配；
- 节点拖动时的关联牵引和回弹；
- 自适应帧率和 Web Worker 布局计算。

### 6. 标签和文档标准化

新文档默认使用 `mytemple/v1` 元数据结构：

```yaml
---
schema: mytemple/v1
title: 文档标题
tags:
  - 标签
domain: 所属领域
created: 2026-07-22
updated: 2026-07-22
status: draft
aliases: []
---
```

旧文档可以在设置中进行标准化：

1. 读取现有 Frontmatter；
2. 保留人工标签和未知字段；
3. 生成修改前后预览；
4. 检查文件是否已被其他程序修改；
5. 用户确认后备份并写入；
6. 重新更新搜索、图谱和 AI 索引。

### 7. AI 规则与安全写入基础

工作区可以创建 `.mytemple/AGENTS.md`，用于规定：

- AI 是否只读；
- AI 可以维护哪些 Markdown 路径；
- 哪些路径和敏感文件永远禁止访问；
- 单次最多允许处理多少文件；
- AI 整理知识库时需要遵守的写作规范。

当前安全原则是：AI 可以检索和生成修改建议，但文档写入必须经过预览与确认。

### 8. AI 知识库主清单

应用会生成 `知识库主清单.json`，其中包含：

- 工作区和文档路径；
- 标题、标签和目录结构；
- 关键词和摘要；
- 双向链接与反向链接；
- 语义概念；
- 未创建链接；
- 图谱节点和关系边；
- 文档哈希及更新时间。

AI 可以先读取主清单筛选候选文档，再按需读取少量原文，避免一次加载整个知识库。

### 9. 多主题与响应式界面

界面支持：

- 明亮、暗色、护眼和图片背景主题；
- 界面字号与正文字号独立调整；
- 多种字体选择；
- 目录宽度拖动；
- 目录隐藏与恢复；
- 设置弹窗响应浏览器窗口尺寸；
- 图谱颜色和背景跟随主题变化。

### 10. 性能与资源控制

为了让浏览器访问保持流畅，项目采用：

- 文件系统变化监听，而不是持续递归轮询；
- `dirtyPaths` 增量文档解析；
- 文档哈希、标签、关键词和图谱缓存复用；
- 目录树懒加载和局部更新；
- 图谱布局使用 Web Worker；
- 图谱节点、邻接表和弹簧边缓存；
- 图谱按拖动、活跃、空闲状态调节帧率；
- 大文档延迟预览；
- 静态 CSS、JavaScript 和 WebP 缓存；
- 知识库版本未变化时不重复写入主清单；
- AI 索引在首次使用时加载，不占用普通编辑启动时间。

## 三、从安装到使用

### 第一步：安装和启动

1. 运行 MyTemple Knowledge 安装程序；
2. 按提示选择安装位置；
3. 确保电脑已安装 Node.js 18 或更高版本；
4. 启动程序后，浏览器会自动打开本地地址；
5. 默认使用说明位于安装目录的 `docs` 文件夹。

项目默认通过本机浏览器访问，不需要为了使用基础功能把文档上传到公共网站。若电脑处于共享网络，应同时使用系统防火墙限制不必要的外部访问。

### 第二步：添加知识库路径

1. 点击左侧“工作路径”；
2. 选择包含 Markdown 文件的本地文件夹；
3. 为路径设置容易理解的名称；
4. 选择是否显示该工作区；
5. 在设置中指定默认工作路径。

建议先添加一个主要知识库，稳定后再增加第二个工作区，避免一开始混合多个用途不同的目录。

### 第三步：建立目录和标签规则

建议从简单结构开始：

```text
00-收集箱/
10-主题知识/
20-技术实践/
30-项目记录/
80-索引/
90-归档/
```

随后统一：

- 文件命名；
- 领域名称；
- 二至五个稳定标签；
- `draft`、`active`、`archived` 状态；
- 双向链接写法。

详细规则参见[[知识库编写规范与AI蒸馏指南]]。

### 第四步：导入和整理旧文档

1. 将旧 Markdown 所在文件夹添加为工作区；
2. 使用全文搜索检查内容是否可以正常读取；
3. 打开旧文档，在设置中选择“标准化当前文档”；
4. 核对 Frontmatter 修改前后差异；
5. 确认后写入；
6. 使用“智能适配标签”预览标签建议；
7. 只应用真正稳定、有分类价值的标签。

对于 `.txt`、`.markdown` 等资料，可以先使用“一键 MD 化”，但转换前应确认文件内容本身符合 Markdown 或纯文本格式。

### 第五步：建立知识关联

在相关文档中使用：

```markdown
- 前置知识：[[容器资源限制]]
- 排查方法：[[Docker容器内存溢出排查]]
- 相关决定：[[生产环境容器内存基线决策记录]]
```

完成后打开图谱：

1. 先选择“当前文档邻域”；
2. 从一层关系开始查看；
3. 根据需要增加深度；
4. 分别开关标签、语义概念和未创建链接；
5. 点击节点返回原文；
6. 对孤立文档补充真正有效的关系，而不是为了连线数量堆砌链接。

### 第六步：启用本地 AI

1. 安装并启动 Ollama；
2. 打开“设置 → 本地 AI 与语义索引”；
3. 保持默认本地地址，或填写自己的 Ollama 地址；
4. 点击“检测连接”；
5. 选择一个 Embedding 模型；
6. 选择一个对话模型；
7. 保存配置并等待后台索引完成；
8. 点击顶部 `AI` 打开知识库问答侧栏。

如果没有配置 Ollama，仍然可以使用来源检索，只是不会生成完整自然语言回答。

### 第七步：开始提问

适合的问题：

```text
关于 Docker 内存溢出的排查记录在哪里？
当前文档中有哪些尚未确认的事项？
哪些文档讨论过 PON 网关 QoS 测试？
这个方案有哪些限制和回退方法？
```

使用回答时：

1. 先看回答结论；
2. 检查每项结论是否带来源编号；
3. 点击来源卡片打开原文；
4. 核对标题、行号和上下文；
5. 重要决定仍由人工确认。

### 第八步：为 AI 建立维护规则

1. 打开设置；
2. 点击“创建规则文件”；
3. 在工作区 `.mytemple/AGENTS.md` 中补充团队或个人规则；
4. 默认保持 `writeMode: confirm`；
5. 只开放需要 AI 维护的 Markdown 路径；
6. 对批量操作先生成预览，再确认应用。

### 第九步：按规范蒸馏文档

将[[知识库编写规范与AI蒸馏指南]]提供给 AI，并明确输入和输出范围。例如：

```text
请按照《知识库编写规范与 AI 蒸馏指南》，将 00-收集箱中的三篇 Docker 记录蒸馏为一篇故障排查指南。

要求保留环境、版本、风险、失败表现和来源；冲突内容放入待确认章节。先输出差异预览，不要直接覆盖原文。
```

### 第十步：建立日常维护节奏

建议：

- 每天：把临时资料放入收集箱；
- 每周：整理标签、未创建链接和孤立文档；
- 每月：归档失效内容，检查高频主题是否需要建立索引页；
- 重要修改：更新 `updated` 日期并核对来源；
- AI 批量处理：先限定文件范围，再检查差异和审计记录。

## 四、典型使用场景

### 技术人员

管理命令、故障排查记录、设备资料、测试方法和项目复盘。遇到问题时直接按语义查找过去的处理经验，并跳转到原始记录。

### 内容创作者

整理选题、素材、长文笔记和观点来源。通过主题页与双向链接复用已有知识，再从知识库中提炼文章结构。

### 项目管理者

保存需求、方案、会议纪要、决定和风险记录。通过绝对日期、状态和来源区分当前结论与历史变化。

### 学习与研究

将课程、论文笔记、术语和实验记录组织为可追踪网络。使用语义问答定位相关内容，但保留原始来源以便复核。

### 本地敏感资料管理

对于不适合上传公共云服务的个人资料，可以使用本地文件夹和本地 Ollama。仍需注意：本地向量索引包含内容特征，应与原文使用相同级别的系统访问保护。

## 五、与普通 Markdown 文件夹相比

| 能力 | 普通文件夹 | MyTemple Knowledge |
|---|---|---|
| Markdown 文件可直接访问 | 支持 | 支持 |
| 多路径统一浏览 | 依赖文件管理器 | 支持工作区 |
| 正文搜索 | 依赖系统或编辑器 | 内置全文搜索 |
| 语义检索 | 不支持 | 支持本地 RAG |
| 回答来源定位 | 不支持 | 文件、标题和行号 |
| 阅读与编辑 | 依赖其他软件 | 内置三模式 |
| 双向链接 | 只作为文本 | 自动识别和跳转 |
| 知识图谱 | 不支持 | 支持筛选和邻域 |
| 元数据标准化 | 人工处理 | 预览、备份后应用 |
| AI 写入保护 | 不适用 | 规则、确认、冲突检查、审计 |
| 数据控制 | 本地 | 本地优先 |

## 六、使用边界说明

为了避免误导，当前版本需要明确以下边界：

- MyTemple Knowledge 是本地知识管理工具，不是多人实时协作平台；
- 不提供默认云同步，用户可以自行配合 Git、NAS 或同步盘；
- AI 回答质量取决于文档质量、Embedding 模型和对话模型；
- Ollama 及模型需要用户自行安装，不随安装包静默下载；
- AI 侧栏目前以检索和问答为主，批量自动维护仍必须经过预览与确认；
- 图谱显示的是文档中可识别的关系，不代表因果关系或绝对正确性；
- 重要技术、医疗、法律和财务结论必须回到原始资料人工核对。

## 七、推广主文案
