# 文档编写与管理规范

这套规范的目标是减少寻找、分类和维护文档的成本。规则应当足够简单，让新文档可以快速创建，也能在几个月后继续读懂和管理。

## 一、文件夹结构

### 按稳定领域分类

优先按照长期不变的职责或主题建立一级文件夹，不要按照临时人员、月份或短期任务创建大量顶层目录。

```text
docs/
├─ 00-索引/
├─ 10-项目/
├─ 20-技术/
├─ 30-流程/
├─ 40-会议与决策/
├─ 90-归档/
└─ README.md
```

数字前缀不是必须的，但可以让常用目录保持稳定顺序。目录层级建议不超过三级。

### 使用索引代替过深目录

当一个目录包含大量文档时，创建主题索引文档，不要继续增加五六层子目录。

## 二、文件命名

推荐使用能够独立表达主题的名称：

```text
网络故障排查.md
知识图谱性能优化.md
2026-07-21-需求评审记录.md
ADR-003-选择本地文件存储.md
```

避免：

```text
新建文档.md
记录2.md
最终版-最新版-修改.md
杂项.md
```

日期适合会议、日报和事件记录；长期维护的知识文档不必在文件名中加入日期。

## 三、标题结构

- 每篇文档只保留一个一级标题 `#`；
- 二级标题 `##` 表示主要章节；
- 三级标题 `###` 表示二级章节下的细分内容；
- 尽量不要跳过标题层级；
- 阅读模式目录会把三级目录默认折叠在二级目录下。

```md
# 文档标题

## 第一部分

### 具体步骤

## 第二部分
```

## 四、标准元数据

推荐在文档开头使用 frontmatter：

```md
---
tags:
  - 项目管理
  - 决策记录
created: 2026-07-21
updated: 2026-07-21
status: active
owner: 姓名或团队
---
```

字段建议：

| 字段 | 用途 | 示例 |
|---|---|---|
| `tags` | 稳定分类 | `项目管理`、`操作手册` |
| `created` | 创建日期 | `2026-07-21` |
| `updated` | 最近实质更新 | `2026-07-21` |
| `status` | 生命周期 | `draft`、`active`、`stable`、`archived` |
| `owner` | 维护责任人 | `网络组` |

系统新建文档会自动写入基础标签模板，可以在首次编辑时替换“待分类”。

## 五、正文写法

### 开头先写用途

标题后使用一段话或引用说明：

```md
> 本文用于指导新成员在十分钟内完成本地知识库初始化。
```

### 结论优先

操作手册先给步骤，决策记录先给结论，复盘文档先给影响。背景资料放在后面，避免读者滚动很久才找到答案。

### 使用可执行清单

```md
- [ ] 确认备份完成
- [ ] 执行配置变更
- [ ] 验证服务状态
- [ ] 更新变更记录
```

### 代码和命令注明环境

```powershell
node server.js
```

命令前说明运行目录、权限要求和预期结果。

## 六、关系管理

### 标签用于分类

每篇文档建议二至五个标签。标签名称保持统一，例如不要同时使用 `知识图谱`、`图谱知识`、`knowledge-graph` 表达同一个概念。

### 双链用于导航

在文档末尾保留“关联文档”章节：

```md
## 关联文档

- 上游：[[需求说明]]
- 方案：[[技术设计]]
- 结果：[[上线复盘]]
```

### 索引用于组织领域

每个重要领域维护一个索引页，将最常用的文档按“入门、流程、参考、复盘”分组。

## 七、不同类型的模板

### 操作手册

```md
---
tags:
  - 操作手册
  - 所属领域
created: 2026-07-21
status: stable
---

# 操作名称

> 适用范围和预期结果。

## 前置条件

## 操作步骤

1. 第一步。
2. 第二步。

## 验证方法

## 回滚方法

## 关联文档
```

### 决策记录

```md
---
tags:
  - 决策记录
  - 所属项目
created: 2026-07-21
status: accepted
---

# 决策：简短结论

## 背景

## 可选方案

## 最终决定

## 影响与风险

## 关联文档
```

### 会议记录

```md
---
tags:
  - 会议记录
  - 所属项目
created: 2026-07-21
status: active
---

# 2026-07-21 会议主题

## 参与人

## 结论

## 待办事项

- [ ] 负责人：任务，截止日期。

## 关联文档
```

## 八、日常管理流程

### 随手记录

先创建文档并记录事实，不要求第一次就整理完美。至少写清标题、用途和来源。

### 当日整理

补充二至五个标签，将“待分类”替换成稳定分类，并添加至少一个关联文档。

### 每周维护

1. 查看“待分类”标签；
2. 检查孤立文档；
3. 处理未创建双链；
4. 合并重复标签；
5. 更新本周核心主题索引。

### 每月归档

将完成且不再频繁修改的项目文档标记为 `archived`，移动到归档目录，但保留必要双链，避免知识网络断裂。

## 九、质量检查清单

- [ ] 文件名能独立说明主题；
- [ ] 只有一个一级标题；
- [ ] 标题层级没有跳跃；
- [ ] 已替换“待分类”；
- [ ] 标签数量在二至五个之间；
- [ ] 开头说明文档用途；
- [ ] 结论、步骤或结果容易找到；
- [ ] 至少有一个有效双链；
- [ ] 没有暴露密码、令牌和隐私信息；
- [ ] 更新日期和状态正确。

## 十、最重要的三条规则

1. 一篇文档只解决一个主要问题；
2. 标签负责分类，双链负责具体关系；
3. 使用索引页组织领域，不使用无限加深的文件夹。
